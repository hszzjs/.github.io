深入分析java IO工作机制

Posted on 2019-06-15 | In 学习

java的I/O大概有80多个类，大概率上是可以分为四组：
按照传输数据的数据格式：字节流与字符流，按照传输数据的传输方式：分为File（磁盘）与Socket（网络）。 所以影响IO的核心问题要么是数据格式影响要么就是传输方式影响。
基于流的操作，前面已经介绍，有一点要注意的就是流最终卸载哪里必须要指定，要么是写到磁盘中要么是写到网络中，写网络事实上也是在写文件，只不过网络进一步处理就是底层操作系统再将数据传送到其他地方而不是本地磁盘。

磁盘I/O的工作机制

数据的操作，前面是说以哪种数据格式来传输数据，还有一个关键问题就是数据写到哪里，有一个主要方式就是将数据持久化到物理磁盘。下面就是数据持久化到物理磁盘的过程。
数据在磁盘的唯一最小描述就是文件，也就是说上层应用程序就只能通过文件来操作磁盘上的数据，文件也是操作系统和磁盘驱动器交互的最小单元。值得注意的是Java中通畅的File并不代表一个真实存在的文件对象，当你指定一个路径描述符时，它就会返回一个代表这个路径相关联的一个虚拟对象，这个可能是一个真实存在的文件或者是一个包含多个文件的目录。
这样去做的原因是，因为大部分情况下，我们并不关心这个文件是否真的存在，而是关心这个文件到底如何操作。
只有当需要真正读取这个文件的时候，才会检查这个文件是不是存在。下图是如何从磁盘读取一段文本字符：

当传入一个文件路径，将会根据这个File对象创建真正读取文件的操作对象，这时将会真正创建一个关联真实存在的磁盘文件的文件描述符FileDescriptor，通过这个对象可以直接控制这个磁盘文件。由于读的的是字符，所以需要将其转码为字节模式，至于如何从磁盘驱动器上读取一段数据，由操作系统帮忙完成。

java Socket的工作机制

Socket就指的是计算机之间完成相互通信的一种抽象功能。大部分情况下Socket使用的是基于TCP/IP的流套接字，是一种稳定的通信协议。 下图是典型的基于Socket的通信的场景：

就主机A的应用程序要能和主机B得到应用程序进行通信，必须经过Socket建立连接，而建立Socket连接必须需要底层TCP/IP协议建立TCP连接，建立TCP连接需要底层IP协议来寻址网络中的主机。网络层使用的IP协议可以帮助根据IP地址寻找目标主机，但是一台主机上可能运行着多个应用程序，如何才能与指定的应用程序通信就是要通过TCP或UDP的地址也就是==端口号==来指定。这样就可以通过一个Socket实例唯一代表一个主机上的应用程序的通信链路了。

建立通信链路

当客户端要与服务端通信，客户端首先要创建一个Socket实例，操作系统将为这个Socket实例分配一个没有被使用的本地端口号，并创建一个包含本地和远程地址和端口号的套接字数据结构，这个数据结构将一直保存在系统中直到这个连接关闭。在创建Socket实例的构造函数正确返回之前，将要进行TCP的三次握手协议，TCP握手协议完成后，Socket实例对象将创建完成，否则将抛出IOException错误。

与之对应的服务端将会创建一个ServerSocket实例，ServerSocket创建比较简单只要指定的端口号没有被占用，一般实例就会创建成功，同时操作系统也会为ServerSocket创建一个底层数据结构，这个数据结构中包含指定监听的端口号和包含监听地址的通配符，通常情况下“”即监听所有地址。之后 *当调用accept()方法时，将进入阻塞状态，等待客户端的请求**。当一个新的请求到来时，将为这个连接创建一个新的套接字数据结构，该套接字数据的信息包含的地址和端口信息正式请求源地址和端口。这个新创建的数据结构将会关联到ServerSocket实例的一个未完成的连接数据结构列表中，注意这时服务端与之对应的Socket实例并没有完成创建，等到与客户端的三次握手完成后，这个服务端的Socket实例才会返回，并将这个Socket实例对应的数据结构从未完成列表中移动到已完成列表中。所以ServerSocket所关联的列表中的每个数据结构，都代表与一个客户端的建立的TCP连接。

当连接已经建立成功，服务端和客户端都会拥有一个Socket实例，每个Socket实例都有一个InputStream和OutputStream（主要原因是网络I/O是以字节流进行传输的），正是通过这两个对象来交换数据。
当Socket对象创建的时候，操作系统就会为InputStream与OutputStream分别分配一定大小的缓冲区，数据的写入和读取都是通过这个缓冲区完成的。写入端将数据写到==OutputStream对应的SendQ队列==中，当队列填满时，数据将被发送到另一端==InputStream的RecvQ队列==中，如果这是RecvQ已经满了，那么OutputStream的write就会阻塞知道可以接受那些数据。那么缓存区的大小以及写入端的速度和读取端的速度非常影响这个连接的数据传输效率，由于可能会发生阻塞，所以网络I/O与磁盘I/O在数据的写入读取时还有一个协调的过程，如果两边同时传送数据就有可能出现死锁。

NIO的工作方式

不管是磁盘IO还是网络IO，数据在写入OutputStream或者从InputStream读取时都有可能会发生阻塞。一旦有线程阻塞就将会失去CPU的使用权（这里的这句话就是说线程阻塞，就是在等，那等的时候CPU为了高效，肯定就不管你了，先去搞其他的，所以就是说失去了CPU的使用权），这在当期那的大规模访问量和有性能要求情况下是不能接受的，虽然当前的网络I/O有一些解决办法，如一个客户端一个处理线程，出现阻塞时只是一个线程阻塞而不会影响其他线程工作，还有为了减少系统线程的开销，采用线程池的办法减少系统创建和回收的成本，但是面对一些问题依旧无法解决，如当前需要大量HTTp长连接的情况，但是并不是每时每刻都在传输数据，那就不能说是同时创建那么多线程来保持连接。即使线程的数量不是问题，仍然有一些问题还是无法避免的，如这种情况，我们想给某些客户端更高的服务优先级，很难通过设计线程的优先级来完成，另外一种情况是，我们需要让每个客户端的请求在服务端可能需要访问一些竞争资源，由于这些客户端是在不同线程中，因此需要同步，而往往要实现这些同步操作要远远比用单线程复杂很多。以上这些情况都说明，我们需要另外一种新的 I/O 操作方式。

NIO的工作机制

NIO的相关类图如下：

可以看到是三个主要类：Channel、Buffer以及Selector，NIO就是通过这三个来实现非阻塞IO的。
Channel就可以看做是一个双向的非阻塞的通道，在通道的两边都可以进行数据的读写操作。
Selector：就是实现了一个线程来管理多个通道（采用了复用与解复用的方式使得一个线程可以管理多个通道，即可以将多个流合并成一个流，或者把一个流分成多个流的方式）。
Buffer：就是缓存区，就是和上面的解释是没有什么不同的，用于缓存数据，可以保存多个channel的数据。 然后就是通过一段NIO代码来理解它们的工作机制：

就调用Selector的静态工厂方法创建一个选择器，创建一个服务端的Channel绑定到一个Socket对象，并把这个通道信道注册到选择器上，把这个通信信道设置为非阻塞模式。然后就可以调用Selector的selectKeys方法来检查已经注册在这个选择器上所有通道信道是否有需要的事件发生，如果有某个事件发生，将会返回所有的SelectionKey，通过这个对象Channel方法就可以取得这个通信信道对象从而可以读取通信的数据，而这里读取的数据是Buffer，这个Buffer就是人类可以控制的缓冲器。 在上面，是将Server端的监听连接请求的时间和处理请求的事件放在一个线程中，但是在实际应用中是放在两个线程中，一个线程专门负责监听客户端的连接请求，而是以阻塞的方式执行的，另一个线程专门是来处理请求，这个专门处理请求的线程才是真正采用NIO的方式，像 Web 服务器 Tomcat 和 Jetty 都是这个处理方式。

Buffer的工作方式

Buffer是如何接受和写出数据？Buffer可以理解是一组基本数据类型的元素列表，通过几个变量来保存这个数据的当前位置状态，也就是有四个索引。
索引 | 说明
—|—
capacity | 缓冲区数组的总长度
position | 下一个要操作的数据元素的位置
limit |缓冲区数组不可操作的下一个元素的位置，limit<=capacity
mark |用于记录当前position的前一个位置或者默认是0

还有一点需要说明，通过 Channel 获取的 I/O 数据首先要经过操作系统的 Socket 缓冲区再将数据复制到 Buffer 中，这个的操作系统缓冲区就是底层的 TCP 协议关联的 RecvQ 或者 SendQ 队列，从操作系统缓冲区到用户缓冲区复制数据比较耗性能，Buffer 提供了另外一种直接操作操作系统缓冲区的的方式即 ByteBuffer.allocateDirector(size)，这个方法返回的 byteBuffer 就是与底层存储空间关联的缓冲区，它的操作方式与 linux2.4 内核的 sendfile 操作方式类似。

NIO必然是实现高速IO。针对的是NIO将最耗时的I/O操作（即填充和提取缓冲区）转移回操作系统，因为极大地提高速度。

流与块的比较

就NIO是面向块来处理数据。但是块是如何存在？注意字符与字节只是传输的数据格式，重点就是传输方式磁盘以及网络IO。磁盘IO没有缓冲这么一说，但是网络也就是Socket IO会有缓冲区以及队列等。 从web开发来讲就是网络IO很容易发生阻塞，而磁盘的IO就是读写的话，读不会阻塞，本机自己写应该遇到阻塞很少，主要还是网络IO。所以==NIO主要面对的就是网络IO==。
在NIO中加入Buffer对象，就是一个创新，面向流的IO中，试讲数据直接写入或读到Stream对象中，而NIO是所有的数据都是用缓冲区处理的，读是读到缓冲区中，写是写到缓冲区中。然后通道的两端必然是缓冲区。
最主要是什么，就是使用了选择器单线程处理，IO就是通过多线程进行多个IO连接读写。但是线程的创建销毁就很麻烦，所以NIO使用选择器来处理，就是说进行轮询管理多个通道，进行处理。
